# Q&A Templates

> UI 설계 요청 시 모호한 부분을 제거하기 위한 질문 템플릿.
> 모든 질문은 **설명 + 추천 + 선택지**를 포함하여 사용자가 쉽게 답변할 수 있도록 한다.
> 사용자가 답변을 잘 모를 경우 "추천" 또는 "추천대로"라고 응답해도 된다.

---

## 질문 형식 규칙

모든 질문은 반드시 아래 형태를 따른다. 형식을 임의로 변경하지 않는다.

```
**Q[N]. [질문]**

> [왜 이 질문이 중요한지 1-2줄 설명]

★ **추천**: [분석 데이터 기반 추천. 추천 이유 1줄]

  a) [옵션] — [간단 설명]
  b) [옵션] — [간단 설명]
  c) [옵션] — [간단 설명]
  d) 직접 입력
```

### 형식 준수 체크리스트

- [ ] 질문 번호는 Q[N] 형식 (Q1, Q2, Q3 ...)
- [ ] 설명은 `>` 인용 블록으로 작성
- [ ] ★ 기호로 추천 항목 표시
- [ ] 선택지는 a) b) c) d) 순서
- [ ] d)는 항상 "직접 입력"으로 고정

---

## "추천대로" 규칙

사용자가 답변에 익숙하지 않거나 빠르게 진행하고 싶을 때를 위한 규칙이다.

### 일괄 적용 트리거

아래 표현 중 하나라도 사용하면 **모든 미답변 질문의 ★ 추천 항목을 일괄 적용**한다.

| 사용자 표현 | 처리 방식 |
|-------------|-----------|
| "추천대로" | 모든 질문 ★ 추천 일괄 적용 |
| "추천대로 해줘" | 모든 질문 ★ 추천 일괄 적용 |
| "알아서 해줘" | 모든 질문 ★ 추천 일괄 적용 |
| "다 추천으로" | 모든 질문 ★ 추천 일괄 적용 |
| "그냥 해줘" | 모든 질문 ★ 추천 일괄 적용 |

### 개별 적용 트리거

특정 질문에 아래 표현을 사용하면 **해당 질문의 ★ 추천만 적용**한다.

| 사용자 표현 | 처리 방식 |
|-------------|-----------|
| "추천" | 해당 질문 ★ 추천 적용 |
| "추천 항목으로" | 해당 질문 ★ 추천 적용 |
| "모르겠어, 추천해줘" | 해당 질문 ★ 추천 적용 |

### 추천 적용 후 응답 형식

추천 항목 적용 시, 아래 형식으로 어떤 선택이 적용되었는지 사용자에게 명확히 알린다.

```
추천 항목을 아래와 같이 적용했습니다:

- Q1. 타겟 사용자 → c) 내부 사용자 (어드민)
- Q2. 레이아웃 관계 → a) 기존 레이아웃 안에 추가
- Q3. 톤앤매너 → a) 현재 프로젝트 스타일 유지
- Q4. 에셋 → c) 없음, 아이콘 활용
- Q5. KPI → b) 3-4개 정도
...

이 설정으로 설계를 시작합니다.
```

## 질문 진행 순서

질문은 반드시 아래 순서를 따른다. 순서를 건너뛰거나 역순으로 진행하지 않는다.

```
1단계: 공통 질문 (Q1~Q4) — 모든 페이지 유형에 해당
       ↓
2단계: 페이지 유형 판별 — Q2 응답 또는 요청 내용 분석
       ↓
3단계: 유형 전용 질문 — 해당 유형의 Q5부터 진행
       ↓
4단계: 설계안 작성 — 모든 답변 취합 후 즉시 시작
```

### 페이지 유형 분류 기준

| 유형 | 판별 키워드 | 전용 질문 범위 |
|------|-------------|----------------|
| Landing / Marketing | 랜딩, 마케팅, 홍보, Hero, 소개 | Q5~Q10 |
| Dashboard | 대시보드, 개요, Overview, KPI, 통계 | Q5~Q8 |
| Form / Settings | 설정, 폼, 입력, 프로필, 환경설정 | Q5~Q7 |
| CRUD | 목록, 관리, 리스트, 테이블, 상세 | Q5~Q7 |
| Auth | 로그인, 회원가입, 인증, 비밀번호 | Q5~Q6 |

### 진행 시 유의사항

- **한 번에 모든 질문을 나열하지 않는다.** 공통 Q1~Q4를 먼저 제시하고, 유형 판별 후 전용 질문을 이어서 진행한다.
- **요청 맥락에서 이미 명확한 항목은 생략한다.** 예: 요청에 "어드민 사용자를 위한"이라고 명시되어 있으면 Q1은 건너뛴다.
- **사용자가 답변을 모를 경우 "추천" 또는 "잘 모르겠어"라고 해도 된다고 안내한다.**
- **모든 질문에 답변이 완료되면 즉시 설계안 작성에 돌입한다.** 불필요한 재확인 절차를 생략한다.

## 공통 질문 (모든 페이지 유형)

모든 페이지 유형에 공통으로 적용되는 질문이다. 반드시 유형 전용 질문 전에 진행한다.

---

**Q1. 이 페이지를 사용하는 주요 타겟은 누구인가요?**

> 타겟 사용자에 따라 톤, 문구, 정보 밀도, 시각적 스타일이 완전히 달라집니다. 예를 들어 B2C는 친근하고 직관적인 UI가, B2B 어드민은 전문적이고 정보 밀도 높은 UI가 효과적입니다.

★ **추천**: 이 프로젝트는 어드민 대시보드 템플릿이므로 내부 사용자(어드민) 기준으로 설계하는 것이 가장 자연스럽고 일관성 있습니다.

  a) 일반 소비자 (B2C) — 친근하고 직관적인 UI, 큰 폰트, 이미지 중심, 낮은 정보 밀도
  b) 기업 고객 (B2B) — 전문적이고 데이터 중심, 테이블/차트 강조, 중간 정보 밀도
  c) 내부 사용자 (어드민) — 효율과 기능 중심, 높은 정보 밀도, 복잡한 인터랙션 허용
  d) 직접 입력

---

**Q2. 새 페이지가 기존 앱 레이아웃 안에 들어가나요, 아니면 독립 페이지인가요?**

> 기존 사이드바+헤더 레이아웃 안에 들어가는지, 별도 독립 페이지(랜딩, 인증 등)인지에 따라 라우팅 구조와 레이아웃 컴포넌트가 근본적으로 달라집니다. 이 결정이 전체 설계 방향을 좌우합니다.

★ **추천**: 대시보드 프로젝트 내부 기능 페이지라면 기존 레이아웃 재사용이 코드 일관성과 개발 효율 모두에서 유리합니다.

  a) 기존 레이아웃(사이드바+헤더) 안에 추가 — 대시보드 내부 기능 페이지
  b) 별도 독립 페이지 — 랜딩, 마케팅, 인증 등 레이아웃 완전 분리 필요
  c) 새로운 레이아웃 그룹 생성 — 별도 구조가 필요한 경우 (예: 풀스크린 에디터, 온보딩 플로우)
  d) 직접 입력

---

**Q3. 전체적인 톤앤매너와 스타일 방향은 어떻게 하시겠어요?**

> 색감, 폰트 무게, 여백, 컴포넌트 스타일(각진/둥근), 전체 분위기를 결정합니다. 기존 프로젝트 디자인 토큰을 재사용할지, 새로운 스타일을 도입할지에 따라 구현 방향이 달라집니다.

★ **추천**: 기존 프로젝트 스타일 유지가 사용자 경험 일관성과 개발 속도 모두에서 가장 유리합니다. 새 스타일 도입은 디자인 시스템 충돌 위험이 있습니다.

  a) 현재 프로젝트 스타일 유지 — 기존 CSS 변수(colors, spacing, radius, shadow) 재사용
  b) 신뢰/전문 — 깔끔한 레이아웃, 블루 계열, 충분한 여백, 모노스페이스 숫자 폰트
  c) 친근/따뜻 — 밝은 컬러, 둥근 모서리(radius-xl), 일러스트 중심, 소프트 섀도우
  d) 프리미엄/다크 — 다크 배경, 세련된 포인트 컬러, 글로우/그라데이션 효과

---

**Q4. 사용할 수 있는 에셋(이미지, 아이콘, 로고 등)이 있나요?**

> 에셋 유무에 따라 레이아웃 구성 방식이 크게 달라집니다. 이미지가 없으면 아이콘과 색상으로 시각적 완성도를 보완해야 하며, 이를 사전에 파악해야 레이아웃 설계 방향이 정해집니다.

★ **추천**: 에셋이 없다면 Lucide 아이콘 + 그라데이션 배경 조합이 가장 범용적이고 빠르게 구현 가능합니다. 이미 프로젝트에 통합된 Lucide를 최대한 활용합니다.

  a) 이미지/일러스트 있음 — 파일 경로 또는 URL을 알려주세요 (public/ 폴더 기준)
  b) 로고만 있음 — 로고 파일 경로를 알려주세요 (SVG 권장)
  c) 없음, 아이콘 활용 — Lucide 아이콘으로 시각적 요소 대체
  d) 없음, 나중에 추가 — 플레이스홀더(회색 박스 + 아이콘)로 진행 후 추후 교체

## Landing / Marketing 전용 질문

Q2 응답이 b) "별도 독립 페이지"이고, 페이지 성격이 랜딩 또는 마케팅 페이지일 때 진행한다.

---

**Q5. 서비스의 핵심 가치 제안(USP)이 무엇인가요?**

> Hero 섹션의 메인 헤드라인 카피가 됩니다. 방문자가 3초 안에 "이게 뭔지" 이해할 수 있어야 전환율이 높아집니다. 명확한 USP는 랜딩 페이지에서 가장 중요한 단일 요소입니다.

★ **추천**: 아직 없다면 "[타겟]을 위한 [핵심기능] 솔루션" 형태로 시작하는 것이 가장 검증된 방식입니다. 서비스 설명을 기반으로 여러 카피 후보를 드릴 수 있습니다.

  a) 이미 있음 — 문구를 그대로 알려주세요 (수정 없이 그대로 사용하겠습니다)
  b) 아이디어만 있음 — 핵심 키워드를 알려주시면 카피로 다듬어드립니다
  c) 없음, 만들어줘 — 서비스 설명을 기반으로 USP 후보 3개를 생성하겠습니다
  d) 직접 입력

---

**Q6. 방문자가 가장 먼저 하길 바라는 행동(CTA)은 무엇인가요?**

> Hero 섹션의 주요 버튼 문구와 배치를 결정합니다. CTA가 명확하고 구체적일수록 클릭률이 높아집니다. 서비스 성격과 판매 사이클에 맞는 CTA 선택이 중요합니다.

★ **추천**: SaaS 서비스는 무료 체험이 데모 요청보다 평균 2-3배 높은 전환율을 기록합니다. 신용카드 불필요 문구를 함께 사용하면 더 효과적입니다.

  a) 무료 체험 시작 — "14일 무료 체험 시작하기" (전환율 최고, B2C/SMB에 적합)
  b) 데모 요청 — "무료 데모 신청하기" (B2B 엔터프라이즈, 고가 제품에 적합)
  c) 회원가입 — "지금 무료로 시작하기" (간결하고 직접적)
  d) 직접 입력

---

**Q7. 신뢰를 높일 소셜 프루프 자료가 있나요?**

> 고객사 로고, 사용자 후기, 수치 등은 방문자의 신뢰를 높여 전환율에 직접적인 영향을 줍니다. Nielsen Norman Group 연구에 따르면 소셜 프루프는 신규 방문자의 결정에 가장 큰 영향을 미칩니다.

★ **추천**: 초기 서비스라면 수치/통계보다 진정성 있는 후기 1-2개가 더 효과적입니다. 아직 없다면 해당 섹션을 생략하고 출시 후 추가하는 것이 자연스럽습니다.

  a) 고객사 로고 있음 — 로고 파일 또는 기업 이름 목록을 알려주세요 (SVG 권장)
  b) 사용자 후기 있음 — 후기 텍스트, 작성자 이름, 직함/회사명을 알려주세요
  c) 수치/통계 있음 — "1,000+ 고객", "99.9% 가동률" 같은 검증된 수치를 알려주세요
  d) 아직 없음, 건너뛰기 — 소셜 프루프 섹션 생략 후 출시 후 추가

---

**Q8. 참고할 만한 벤치마크 사이트나 디자인 방향이 있나요?**

> 구체적인 레퍼런스가 있으면 디자인 방향 합의가 빠르고 수정 횟수가 줄어듭니다. "이런 느낌"을 사전에 정의해두면 결과물에 대한 기대치 불일치를 방지할 수 있습니다.

★ **추천**: 없으면 Linear, Vercel, Notion 스타일의 현대적이고 깔끔한 SaaS 랜딩 디자인으로 진행합니다. 화려함보다 명확함이 우선입니다.

  a) 참고 사이트 있음 — URL을 알려주세요 (여러 개도 가능, 각 사이트의 좋은 점 설명 환영)
  b) 비슷한 경쟁 서비스 있음 — 서비스 이름을 알려주세요 (차별점도 함께 알려주시면 좋습니다)
  c) 없음, 표준 스타일로 — 현대적인 SaaS 랜딩 표준 디자인 (Linear/Vercel 스타일)
  d) 직접 입력

---

**Q9. Hero 섹션에 사용할 비주얼이 있나요?**

> Hero 섹션은 랜딩 페이지에서 시각적 임팩트가 가장 큰 영역으로, 첫인상을 결정합니다. 이미지 유무에 따라 레이아웃이 텍스트+이미지 분할형인지, 텍스트 중심인지가 결정됩니다.

★ **추천**: 실제 제품 스크린샷이 없다면 그라데이션 배경 + 대시보드 UI 목업이 신뢰감과 시각적 완성도를 동시에 높이는 가장 효과적인 방식입니다.

  a) 제품 스크린샷/목업 있음 — 파일 경로 또는 URL을 알려주세요
  b) 사진/일러스트 있음 — 파일 경로 또는 URL을 알려주세요
  c) 없음, 그라데이션 + 대시보드 스크린샷 — 실제 제품 UI를 브라우저 프레임에 담아 사용
  d) 없음, 추상적 배경 — 기하학 패턴/도형/그라데이션으로 대체

**Q10. 랜딩 페이지 상단 네비게이션 구성은 어떻게 할까요?**

**Q5. 대시보드 상단에 표시할 핵심 지표(KPI)는 무엇인가요?**

**Q6. 데이터를 차트로 시각화할 필요가 있나요?**

★ **추천**: 라인 차트 1개(시간별 추이) + 바 차트 1개(항목별 비교) 조합이 가장 범용적이고 정보 전달이 명확합니다. 이 프로젝트는 Recharts가 설치되어 있어 바로 사용 가능합니다.

  a) 라인 차트 — 시간에 따른 변화 추이 (매출, 방문자, 가입자 등)
  b) 바 차트 — 카테고리별 또는 기간별 비교 (채널별 매출, 월별 신규 가입 등)
  c) 둘 다 — 라인 + 바 조합으로 추이와 비교를 모두 표시
  d) 차트 불필요 — 숫자 카드와 테이블만으로 충분

  a) 필요 — 어떤 데이터를 목록으로 보여줄지 알려주세요 (예: 최근 주문, 최근 가입자, 최근 결제)
  b) 간단한 목록으로 — 5-10개 항목의 최근 활동 피드 형태 (아이콘 + 텍스트 + 시간)
  c) 불필요 — KPI 카드와 차트만으로 충분

**Q8. 데이터 갱신 방식은 어떻게 할까요?**

★ **추천**: 대부분의 어드민 대시보드는 페이지 로드 시 1회 조회로 충분합니다. 우측 상단에 "마지막 갱신 시간 + 새로고침 버튼"을 추가하면 신뢰감을 줄 수 있습니다.

  c) 수동 갱신 — 새로고침 버튼 클릭 시 갱신, 마지막 갱신 시간 표시
  d) 페이지 로드 시 1회 — 가장 단순하고 안정적 (일반 비즈니스 대시보드 표준)

**Q6. 설정 변경 시 언제 저장되어야 하나요?**

  b) 수동 저장 — 하단 고정 "변경사항 저장" 버튼 (이메일, 비밀번호 등 중요 정보에 적합)

★ **추천**: 필수 필드 표시 + 이메일/전화번호 형식 검증 + 저장 시점 일괄 검증이 기본입니다. react-hook-form + zod 조합으로 빠르게 구현 가능합니다.

  b) 엄격 — 타이핑 중 실시간 유효성 검증, 상세 에러 메시지, 비밀번호 강도 표시기

**Q6. 회원가입 시 수집할 정보는 무엇인가요?**

## 설계안 작성 기준

모든 질문에 답변이 완료되면 아래 기준으로 설계안을 작성한다.

### 설계안 포함 항목

```markdown
## [페이지명] 설계안

### 페이지 개요
- 유형: [Landing/Dashboard/Form/CRUD/Auth]
- 레이아웃: [기존 레이아웃 내부 / 독립 페이지]
- 타겟: [B2C / B2B / 어드민]

### 파일 구조
[생성할 파일 목록과 역할]

### 컴포넌트 구성
[주요 섹션별 컴포넌트 트리]

### 데이터 구조
[필요한 타입, 목 데이터 구조]

### 구현 순서
1. [첫 번째로 구현할 것]
2. [두 번째로 구현할 것]
...

### 기술 스택
- [사용할 라이브러리/컴포넌트 목록]
```

### 설계안 작성 규칙

- 답변 수집 후 불필요한 재확인 없이 즉시 설계안 작성을 시작한다.
- 불확실한 부분은 합리적인 가정을 하고 주석으로 명시한다.
- 구현 순서는 가장 핵심적인 부분(페이지 레이아웃 + 기본 데이터 표시)을 먼저 배치한다.

## 유형별 추천 컴포넌트 스택

각 페이지 유형에서 자주 사용되는 컴포넌트 조합을 미리 참조한다. 설계안 작성 시 아래 스택을 기반으로 제안한다.

### Landing / Marketing

| 섹션 | 사용 컴포넌트 | 비고 |
|------|--------------|------|
| Navbar | `<nav>` + shadcn `Button` | sticky 포지션, 스크롤 시 배경 전환 |
| Hero | 그리드 레이아웃 + `Badge` + `Button` x2 | 헤드라인 + 서브카피 + CTA 2개 (주/보조) |
| Features | CSS Grid 3열 + `Card` + Lucide 아이콘 | 아이콘 + 타이틀 + 설명 |
| Social Proof | Flex + 로고 목록 또는 `Card` (후기) | 그레이스케일 로고 처리 권장 |
| Pricing | CSS Grid + `Card` + `Badge` (추천) | 가장 인기 있는 플랜 강조 표시 |
| CTA Section | 다크 배경 섹션 + `Button` | 페이지 하단 재유도 |
| Footer | Grid 4열 + 링크 목록 | 반응형에서 1열로 변경 |

### Dashboard

| 섹션 | 사용 컴포넌트 | 비고 |
|------|--------------|------|
| KPI Cards | shadcn `Card` x4 + Lucide 아이콘 | 변동률은 녹색/빨간색 텍스트 + 화살표 아이콘 |
| Line Chart | Recharts `LineChart` + `ResponsiveContainer` | 기간 필터(7일/30일/90일) 탭 포함 |
| Bar Chart | Recharts `BarChart` | X축 라벨 회전 처리 |
| Recent Table | shadcn `Table` + `Badge` (상태) | 5-8행, "전체 보기" 링크 |
| Header | 기존 `PageHeader` 컴포넌트 재사용 | 날짜 범위 선택기 우측 배치 |

### Form / Settings

| 섹션 | 사용 컴포넌트 | 비고 |
|------|--------------|------|
| 섹션 구분 | shadcn `Separator` + 섹션 타이틀 | h3 + 설명 텍스트 + Separator |
| 텍스트 필드 | shadcn `Input` + `Label` + `FormMessage` | react-hook-form 연동 |
| 드롭다운 | shadcn `Select` | Radix UI 기반, 접근성 자동 처리 |
| 토글 | shadcn `Switch` | 자동 저장 방식에 사용 |
| 파일 업로드 | shadcn `Input type="file"` + 드래그앤드롭 | 아바타/로고 업로드에 사용 |
| 저장 버튼 | shadcn `Button` (default) + 로딩 상태 | sticky bottom 또는 섹션 하단 |
| 에러 메시지 | `FormMessage` (shadcn) | zod 스키마 연동 자동 표시 |

### CRUD

| 섹션 | 사용 컴포넌트 | 비고 |
|------|--------------|------|
| 검색 | shadcn `Input` + Lucide `Search` 아이콘 | Debounce 300ms 적용 |
| 필터 | shadcn `Select` 또는 `DropdownMenu` | 다중 필터는 Popover 내부에 배치 |
| 테이블 | TanStack Table + shadcn `Table` | 정렬, 페이지네이션 내장 |
| 상태 배지 | shadcn `Badge` (variant 커스텀) | 상태별 색상 variant 정의 |
| 행 액션 | shadcn `DropdownMenu` (점 세 개 버튼) | 수정/복제/삭제 메뉴 |
| 생성/수정 폼 | shadcn `Dialog` 또는 `Sheet` | 단순 폼은 Dialog, 복잡한 폼은 Sheet |
| 삭제 확인 | shadcn `AlertDialog` | 되돌릴 수 없는 작업에 반드시 사용 |
| 페이지네이션 | shadcn `Pagination` | 또는 "더 보기" 무한 스크롤 |

### Auth

| 섹션 | 사용 컴포넌트 | 비고 |
|------|--------------|------|
| 레이아웃 | 좌우 분할 또는 중앙 카드 | 좌측 브랜드 비주얼 + 우측 폼 |
| 로그인 폼 | `Card` + `Input` + `Button` | react-hook-form + zod 유효성 검증 |
| 소셜 버튼 | `Button` (outline) + 브랜드 아이콘 | Google SVG 아이콘 인라인 사용 |
| 구분선 | `<div>` + `Separator` + "또는" 텍스트 | 소셜/이메일 구분 |
| 비밀번호 필드 | `Input` + 보기/숨기기 토글 버튼 | `type="password"` ↔ `type="text"` |
| 에러 상태 | shadcn `Alert` (destructive) | 로그인 실패 메시지 표시 |

## 자주 묻는 질문 (FAQ)

### Q. 질문을 몇 개까지 해야 하나요?

유형에 따라 다르다. Auth 페이지는 공통 4개 + 전용 2개로 총 6개가 최소이며, Landing 페이지는 최대 10개까지 진행한다. 사용자가 이미 알려준 정보는 건너뛰어 총 질문 수를 최소화한다. 목표는 설계에 필요한 정보를 수집하는 것이지, 모든 질문을 강제로 진행하는 것이 아니다.

### Q. 사용자가 질문 중간에 "그냥 만들어줘"라고 하면 어떻게 하나요?

남은 질문 전부의 ★ 추천 항목을 일괄 적용하고, 어떤 선택이 적용되었는지 요약 후 즉시 설계안 작성을 시작한다. 사용자의 시간을 존중하는 것이 우선이다.

### Q. 추천 항목이 프로젝트 상황과 맞지 않는 경우는 어떻게 하나요?

추천은 일반적인 베스트 프랙티스 기반이다. 프로젝트의 기존 패턴, 기술 스택, 또는 비즈니스 요구사항이 추천과 다를 경우, 프로젝트 상황에 맞는 선택을 우선한다. 추천 이유를 설명하되 강제하지 않는다.

### Q. 새로운 페이지 유형이 생기면 어떻게 하나요?

위 5가지 유형에 해당하지 않는 경우, 공통 Q1~Q4를 진행한 후 해당 유형의 특성에 맞는 질문을 즉흥적으로 구성한다. 이 문서에 없는 유형(예: 온보딩 플로우, 결제 페이지, 이메일 에디터)은 별도 섹션으로 추가될 수 있다.

### Q. 사용자가 선택지 외의 답변을 하면 어떻게 하나요?

d) 직접 입력과 동일하게 처리한다. 사용자가 "c)인데 색상은 파란색으로" 같이 선택지를 변형해서 답변해도 그대로 수용한다. 선택지는 가이드이지 제약이 아니다.

## 페이지 유형별 섹션 구성 가이드

각 유형에서 표준적으로 포함되는 섹션 목록이다. 질문 답변에 따라 일부 섹션을 추가하거나 생략한다.

### Landing 페이지 표준 섹션 순서

```
1. Navbar (고정 상단)
2. Hero (USP + CTA + 비주얼)
3. Social Proof (로고 또는 수치) — 선택
4. Features (핵심 기능 3-6개)
5. How It Works (사용 흐름) — 선택
6. Testimonials (사용자 후기) — 선택
7. Pricing (요금제) — 선택
8. FAQ (자주 묻는 질문) — 선택
9. CTA Section (하단 재유도)
10. Footer
```

### Dashboard 페이지 표준 섹션 순서

```
1. Page Header (제목 + 날짜 필터 + 새로고침)
2. KPI Cards (3-4개 가로 배치)
3. Charts (라인 차트 + 바 차트 또는 도넛)
4. Recent Activity Table (최신 데이터 목록)
5. 보조 위젯 (빠른 액션, 알림, 진행률 등) — 선택
```

### Form/Settings 페이지 표준 섹션 순서

```
1. Page Header (제목 + 설명)
2. 섹션 탭 또는 사이드 네비게이션 (항목이 많은 경우)
3. 각 설정 섹션 (Separator + 섹션 타이틀 + 필드 그룹)
4. 저장 버튼 (sticky bottom 또는 각 섹션 하단)
5. 위험 구역 (계정 삭제 등) — 최하단 배치, AlertDialog 사용
```

### CRUD 페이지 표준 섹션 순서

```
1. Page Header (제목 + 생성 버튼)
2. 검색 + 필터 바
3. 데이터 테이블 (정렬 + 상태 배지 + 행 액션)
4. 페이지네이션
5. 생성/수정 Dialog 또는 Sheet (테이블 위에 오버레이)
6. 삭제 확인 AlertDialog
```

### Auth 페이지 표준 섹션 순서

```
1. 브랜드 로고 (상단 중앙 또는 좌측 패널)
2. 소셜 로그인 버튼 (위에 배치, 첫 번째 CTA)
3. 구분선 ("또는 이메일로 계속")
4. 이메일 + 비밀번호 폼
5. 보조 링크 (비밀번호 찾기, 회원가입/로그인 전환)
6. 약관 동의 (회원가입 시) — 선택
```

## 체크리스트: 설계안 작성 전 확인

설계안 작성을 시작하기 전, 아래 항목을 모두 확인한다.

- [ ] **타겟 사용자** 확정 (Q1)
- [ ] **레이아웃 관계** 확정 (Q2) — 기존 레이아웃 내부 vs 독립 페이지
- [ ] **페이지 유형** 판별 완료 — Landing / Dashboard / Form / CRUD / Auth
- [ ] **유형 전용 질문** 모두 답변 완료

### 기술 스택 확인 항목

- [ ] `package.json`에서 사용 가능한 라이브러리 확인
  - shadcn/ui 컴포넌트 목록
  - 차트 라이브러리 (Recharts 등)
  - 폼 라이브러리 (react-hook-form, zod)
  - 테이블 라이브러리 (TanStack Table)
- [ ] 기존 레이아웃 컴포넌트 파악 (`app/` 또는 `components/layouts/`)
- [ ] 기존 디자인 토큰 파악 (`globals.css` CSS 변수)

### 설계안 품질 확인 항목

- [ ] 모든 답변이 설계안에 반영되었는가
- [ ] 기존 프로젝트 파일 구조 컨벤션을 따르는가
- [ ] 반응형 브레이크포인트(모바일/태블릿/데스크톱)가 고려되었는가
- [ ] 접근성(ARIA, 키보드 내비게이션)이 고려되었는가
- [ ] 로딩/에러/빈 상태(Empty State)가 설계에 포함되었는가
